# 任务：[FEATURE NAME]

**输入**：来自 `/specs/[###-feature-name]/` 的设计文档  
**前置条件**：plan.md（必填）、spec.md（编写用户故事所需）、research.md、data-model.md、contracts/

**测试**：下方示例包含测试相关任务。只有在功能规格明确要求时才添加测试任务。

**组织方式**：任务按用户故事分组，确保每个故事都能独立实现与测试。

## 格式：`[ID] [P?] [Story] 描述`
- **[P]**：可并行执行（不同文件、无依赖）
- **[Story]**：对应的用户故事（例如 US1、US2、US3）
- 描述中请写明精确的文件路径

## 路径约定
- **单项目**：仓库根目录下的 `src/`、`tests/`
- **Web 应用**：`backend/src/`、`frontend/src/`
- **移动端**：`api/src/`、`ios/src/` 或 `android/src/`
- 下方示例默认按单项目结构，请根据 plan.md 的实际结构进行调整

## 依赖与执行顺序

### 阶段之间的依赖

- **环境搭建（阶段 1）**：无依赖——可立即开始
- **基础设施（阶段 2）**：依赖阶段 1，阻塞所有用户故事
- **用户故事（阶段 3+）**：均依赖基础设施完成
  - 具备人手时可并行推进
  - 或按优先级顺序依次完成（P1 → P2 → P3）
- **打磨阶段（最终阶段）**：依赖所有目标用户故事完成

### 用户故事之间的依赖

- **用户故事 1（P1）**：基础设施完成后即可开始，不依赖其他故事
- **用户故事 2（P2）**：基础设施完成后即可开始，可与 US1 集成但应保持独立可测
- **用户故事 3（P3）**：基础设施完成后即可开始，可与 US1/US2 集成但应保持独立可测

### 每个用户故事内部

- 若包含测试，必须先编写并确保实现前测试失败
- 先完成模型，再完成服务
- 服务完成后再实现端点
- 核心实现完成后再处理集成
- 完成一个故事后再进入下一个优先级

### 并行机会

- 所有标记为 [P] 的环境搭建任务可以并行执行
- 基础设施阶段中标记为 [P] 的任务可以并行执行
- 基础设施完成后，如团队人手允许，用户故事可并行推进
- 用户故事中标记为 [P] 的测试可以并行执行
- 同一故事内标记为 [P] 的模型任务可以并行进行
- 不同用户故事可由不同成员分别负责并并行推进

## 实施策略

### MVP 优先（仅完成用户故事 1）

1. 完成阶段 1：环境搭建
2. 完成阶段 2：基础设施（关键——阻塞所有故事）
3. 完成阶段 3：用户故事 1
4. **暂停并验证**：独立测试用户故事 1
5. 如具备条件即可发布/演示

### 增量交付

1. 完成环境搭建 + 基础设施 → 基础能力就绪
2. 添加用户故事 1 → 独立测试 → 发布/演示（MVP！）
3. 添加用户故事 2 → 独立测试 → 发布/演示
4. 添加用户故事 3 → 独立测试 → 发布/演示
5. 每个故事都能在不破坏已有能力的前提下增加价值

### 团队并行策略

假如有多名开发者：

1. 团队合力完成环境搭建和基础设施
2. 基础设施完成后：
   - 开发者 A：用户故事 1
   - 开发者 B：用户故事 2
   - 开发者 C：用户故事 3
3. 各故事独立完成并集成

## 备注

- 标记为 [P] 的任务表示没有依赖、可在不同文件中并行执行
- [Story] 标签用于映射任务与具体用户故事，便于追踪
- 每个用户故事都应当能够独立完成与测试
- 在实现前需确认测试先编写并失败
- 建议每完成一个任务或合理分组后提交一次
- 可以在任一检查点暂停，以独立验证对应故事
- 避免出现：描述模糊、修改同一文件产生冲突、跨故事的强耦合依赖
