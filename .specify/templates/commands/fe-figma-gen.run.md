## 用户输入

```text
$ARGUMENTS
```

请提供需求描述、目标（页面/组件）、Figma 文件或节点链接，以及任何特殊注意事项（如仅生成骨架、保持 SSR、禁止新增依赖）。

## 前置条件

- `.specify/memory/fe-figma-gen.md` 已由 `/speckit.fe-figma-gen.wizard` 或 `/speckit.fe-figma-gen.scan` 填写完成；缺失时提示先补全约束。
- 目标包/应用具备构建和测试脚本（参考 `.specify/memory/fe-rule.md` 或用户补充说明）。
- 已配置并可访问内部 Figma 分块伪代码工具与官方 Figma 代码获取工具（调用方式由项目默认提供，无需在模板中填写）。

## 目标

基于 Figma 设计与项目约束，完成页面或组件的落地实现，严格复用共享资源并生成必要的伴随文件（样式、测试、文档等），并在结束时给出验证建议。

## 执行步骤

1) **初始化与运行记录定位**
   - 从仓库根目录运行 `scripts/powershell/check-prerequisites.ps1 -Json -PathsOnly` 并解析其 JSON 输出，至少获取 `FEATURE_DIR`、`BRANCH`、`IMPL_PLAN` 的绝对路径。若参数包含单引号（如 "I'm Groot"），请使用 `'I'\''m Groot'` 或双引号包裹。
   - 定义 `RUNBOOK_DIR = FEATURE_DIR/runbooks` 与 `RUNBOOK_FILE = RUNBOOK_DIR/fe-figma-gen.md`（统一使用绝对路径）。如目录/文件不存在则先创建目录，并拷贝 `.specify/templates/fe-figma-gen-run-template.md` 用作初始内容。
   - 读取现有运行文档以便后续增量更新，保留既有的历史记录与上下文。

2) **加载约束**
   - 读取 `.specify/memory/fe-figma-gen.md`，获取：框架与 UI 库、目录规范、共享色值 Token 文件、Figma 标签含义、命名约束与禁止事项。
   - 读取 `.specify/memory/fe-rule.md`（若存在），补充工程规范、测试策略等上下文。

3) **解析需求**
   - 将用户输入整理为结构化需求（目标类型、页面/组件名称、场景、交互、数据依赖、Figma 链接）。
   - 如果缺少关键信息（例如 Figma 节点、目标目录），向用户确认后再继续。

4) **采集 Figma 伪代码**
   - 调用项目提供的 Figma 分块伪代码工具（调用 fuxi-figma 的 tool），传入 Figma 链接及所需参数，注意 depth 不要传信息，我需要整棵树的信息，生成分块描述，如果返回内容为空或者工具调用失败则直接进入下一步骤。
   - 验证输出遵循标签规则（`use-*`/`create-*` 标签）；若缺失或冲突，记录问题并提示用户修正设计标注。

5) **调用官方导出工具**
   - 紧接着调用官方 Figma 代码获取工具（调用 figma 的 tool），对已确认的节点生成参考代码或模板。
   - **组件规划策略**（根据伪代码是否可用采用不同方式）：
     * **有伪代码时**：将官方导出结果与伪代码对照，识别：需复用的共享布局/组件（`use-*` 标签）、需新建的目录（`create-*` 标签）、第三方组件调用点。
     * **无伪代码时**（工具失败或返回空）：基于以下依据自主规划组件拆分：
       - 官方导出结果的层级结构（Frame/Component/Instance 关系）
       - 项目约束中定义的组件命名与目录规范（参考上文已经读取的约束 `.specify/memory/fe-figma-gen.md`）
       - 现有共享组件库的复用可能性（检查相似布局、重复模式）
       - 通用拆分原则：单一职责、可复用性、合理的组件粒度
   - 调用官方 figma 获取变量的 tool 获取设计稿 Design Token
   - 对导出结果中的样式变量、色值、尺寸、排版等字段，尽可能匹配现有代码库中的共享常量或 Design Token，避免重复定义。

6) **生成实施计划**
   - 基于标签映射整理待执行操作：文件创建/修改、共享资源复用、依赖引入、测试/文档补充。
   - 所有路径必须与模板中的目录约束一致；若需新增路径，先在计划中解释原因。
   - 列出需运行的脚本（lint/test/build）与潜在风险。
   - 输出计划并询问："是否根据以上计划开始实施？（是/否）"，在获得确认前可能会有多轮对话进行进一步的方案调整，如果用户给出修改意见你要重新询问是否开始实施？（是/否），在最终确认前不得写入磁盘。

7) **实施与生成代码**
   - 按计划执行，优先处理共享资源（布局/组件）→ 页面/局部组件 → 测试/故事/文档。
   - 复用或创建组件时，严格遵循命名与目录规范；在新建共享资源时同步更新索引或导出文件。
   - 若需要编写样式或交互逻辑，参考模板中指定的 UI 库和样式体系。
   - 若约束中提供公用色值或 Design Token 文件，优先复用其中定义的颜色变量，并在新建 Token 时保持命名约定。
   - 对伪代码中的标签做落地记录，如 `use-component-share/CommonTag` 对应 import 路径。
   - 注意最终的执行计划中需要考虑到复用的部分的具体计划。
   - 对于无法自动完成的步骤（例如待确认的 API），在输出中标记并请求人工处理。

8) **校验与自测**
   - 运行模板或仓库推荐的命令（例如 `pnpm lint`、`pnpm test --filter <workspace>`、`pnpm --filter <workspace> build`）。若环境限制无法执行，提供命令与预期结果供用户手动运行。
   - 校验确认使用的 css 变量在项目中是否真的存在，如果不存在回退为常量的定义
   - 确保所有新文件通过格式化及类型检查；引用的共享组件存在并与最新实现兼容。

9) **沉淀执行记录**
   - 以 `RUNBOOK_FILE` 为目标，更新 `.specify/templates/fe-figma-gen-run-template.md` 定义的结构：
     * 同步 "基础信息" 中的功能目录（`FEATURE_DIR` 绝对路径）与最新更新时间（使用 ISO 8601 时间戳）。
     * 以当前运行整理 "## 当前执行摘要"，填充需求目标、关键输入、复用组件/样式、生成的代码与文档、已执行的校验、遗留问题/待确认项。
     * 在 "## 历史记录" 顶部追加一条以 `### YYYY-MM-DD HH:mm - <目标名称>` 形式记录的子章节，包含：输入概要、复用项、新增文件/目录、验证结果、后续动作。保持最新记录置顶。
   - 若文件原有内容包含人工补充说明，需保留并在更新时避免覆盖；仅增量更新模板定义区域。
   - 根据这次运行的对话检查`.specify/memory/fe-figma-gen.md`是否需要修订调整，如果需要请你修订调整（直接修改原来的内容即可）

10) **汇报**
   - 列出新增/修改文件、复用的共享资源、创建的布局/组件，以及两次 Figma 工具调用的关键产出（伪代码概要、导出片段）。
   - 汇总 `RUNBOOK_FILE` 路径、此次写入的重点摘要，并提醒可在下次运行前先阅读该运行记录。
   - 记录任何未解决的风险或需要用户后续处理的事项。
   - 建议下一步操作（提交前的最终命令、设计回传、代码评审重点）。

## 注意事项

- 遵循模板中的路径与命名；禁止随意创建新目录或引入未获批准的依赖。
- 所有引用路径使用绝对路径或符合工作区约定的别名。
- 若 Figma 标注与代码约束冲突，优先寻求用户澄清，而非自行动态调整。
- 生成的代码需与现有 lint/prettier 规则兼容；必要时调用格式化工具。
