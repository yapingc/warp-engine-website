## 用户输入

```text
$ARGUMENTS
```

在继续前，若输入非空，必须先审视用户输入。

## 目标

在 `/tasks` 已经成功生成完整 `tasks.md` 之后，识别三份核心产物（`spec.md`、`plan.md`、`tasks.md`）之间的不一致、重复、歧义与定义不足。

## 操作约束

**严格只读**：禁止修改任何文件。输出结构化分析报告即可。若要提供整改建议，仅能作为可选方案，且必须等待用户明确批准后才允许运行后续编辑命令（由用户手动触发）。

**宪章优先**：分析范围内不得违背项目宪章（`/memory/constitution.md`）。若发现冲突，自动视为 **CRITICAL**，必须通过调整规格、计划或任务来解决，而不是弱化、曲解或无视宪章原则。若需要修改宪章本身，必须在 `/analyze` 命令之外的流程中显式进行。

## 执行步骤

### 1. 初始化分析上下文

从仓库根目录运行 `{SCRIPT}` 一次，解析 JSON，获取 FEATURE_DIR 与 AVAILABLE_DOCS。推导以下绝对路径：

- SPEC = FEATURE_DIR/spec.md
- PLAN = FEATURE_DIR/plan.md
- TASKS = FEATURE_DIR/tasks.md

若任一必需文件缺失，需立即报错并提示用户先运行缺失的前置命令。
当参数中包含单引号（如 "I'm Groot"）时，需使用转义写法，例如 `'I'\''m Groot'`，或改用双引号 `"I'm Groot"`。

### 2. 渐进式加载产物

仅加载分析所需的最小上下文：

**来自 spec.md：**

- Overview/Context
- Functional Requirements
- Non-Functional Requirements
- User Stories
- Edge Cases（若存在）

**来自 plan.md：**

- 架构/技术栈选择
- 数据模型引用
- 阶段划分
- 技术约束

**来自 tasks.md：**

- 任务 ID
- 描述
- 阶段分组
- 并行标记 [P]
- 引用的文件路径

**来自宪章：**

- 加载 `/memory/constitution.md` 以校验原则

### 3. 构建语义模型

创建内部表示（输出中不要直接粘贴原文）：

- **需求清单**：将每条功能/非功能需求生成稳定键名（依据命令式短语，例如 "User can upload file" → `user-can-upload-file`）
- **用户故事/动作清单**：拆分用户动作并附上验收标准
- **任务覆盖映射**：将每个任务与一条或多条需求/故事关联（根据关键词或显式 ID/短语）
- **宪章规则集**：提取原则名称与 MUST/SHOULD 类的规范描述

### 4. 检测流程（聚焦高价值信号）

聚焦高价值发现，总数最多 50 条；超出部分须汇总提及。

#### A. 重复检测

- 找出近似重复的需求
- 标记语义弱的描述，建议收敛

#### B. 歧义检测

- 标记缺乏可量化依据的模糊形容词（fast、scalable、secure、intuitive、robust 等）
- 标记未处理的占位符（TODO、TKTK、???、`<placeholder>` 等）

#### C. 定义不足

- 仅有动词、缺乏对象或可衡量结果的需求
- 缺少验收标准对应的用户故事
- 引用未在 spec/plan 中定义的文件或组件的任务

#### D. 宪章对齐

- 任意需求或计划内容与 MUST 级原则冲突
- 宪章要求的章节或质量关口缺失

#### E. 覆盖缺口

- 未被任何任务覆盖的需求
- 没有关联需求/故事的任务
- 未被任务体现的非功能需求（如性能、安全）

#### F. 不一致

- 术语漂移（同一概念出现不同名称）
- 计划中提到的数据实体未在规格中出现（或反之）
- 任务排序矛盾（例如基础设置未完成前就安排集成任务，且无依赖说明）
- 需求互相冲突（如一个写 Next.js，另一个要求 Vue）

### 5. 严重性划分

使用以下优先级策略：

- **CRITICAL**：违反宪章 MUST、缺少核心文档、关键需求完全无覆盖
- **HIGH**：重复/冲突需求、模糊的安全/性能要求、不可验证的验收标准
- **MEDIUM**：术语漂移、非功能需求缺乏任务覆盖、边界条件描述不足
- **LOW**：措辞或样式改进、对执行顺序无影响的轻度冗余

### 6. 生成紧凑分析报告

输出 Markdown 报告（禁止写文件），结构如下：

## 规格分析报告

| ID | 分类 | 严重级别 | 位置 | 摘要 | 建议 |
|----|------|----------|------|------|------|
| A1 | 重复 | 高 | spec.md:L120-134 | 存在两条含义相近的需求…… | 合并并保留表述更清晰的一条 |

（每条发现对应一行；ID 前缀使用分类首字母。）

**覆盖率汇总表：**

| 需求键 | 是否有任务 | 关联任务 ID | 备注 |
|--------|------------|-------------|------|

**宪章对齐问题：**（若存在）

**未映射任务：**（若存在）

**指标：**

- 总需求数
- 总任务数
- 覆盖率（至少关联 1 项任务的需求占比）
- 歧义项数量
- 重复项数量
- CRITICAL 问题数量

### 7. 提供下一步行动建议

在报告末尾给出简洁的“下一步行动”：

- 若存在 CRITICAL 问题：建议在 `/implement` 前优先解决
- 若仅剩 LOW/MEDIUM：可继续推进，同时给出改进意见
- 明确推荐命令，例如：“运行 /specify 进行细化”、“运行 /plan 调整架构”、“手动更新 tasks.md 以补充 `performance-metrics` 覆盖”

### 8. 提出整改方案

询问用户："需要我为排名前 N 的问题建议具体整改步骤吗？"（不要自动执行任何修改。）

## 操作原则

### 上下文效率

- **高信噪比**：聚焦可执行发现，而非穷举文档
- **渐进式加载**：按需读取内容，避免一次性加载全部
- **节省输出**：发现表格不超过 50 行，超出部分汇总概述
- **结果可复现**：在输入不变的情况下多次运行应得出一致的 ID 与计数

### 分析准则

- **绝不修改文件**
- **绝不臆造缺失章节**（缺什么就如实报告）
- **优先处理宪章冲突**（一律视为 CRITICAL）
- **以示例说明**（引用具体实例，而非泛泛规则）
- **若无问题也需明确说明**，并提供覆盖率等统计
