## 用户输入

```text
$ARGUMENTS
```

用户输入（如果有）可指定本次需求上下文（任务编号、影响范围、优先级）或补充约束（例如"坚守 SSR"、"避免新增依赖"）。

## 前置条件

- `.specify/memory/fe-rule.md` 必须存在且为最新版本（如不确定，请先运行 `/speckit.fe-rule.scan` 或 `/speckit.fe-rule.wizard`）。
- 若项目使用计划/任务产物（如 `.specify/memory/plan.md`、`plan.md`、`tasks.md`），需确保可读取，如果相关文件不存在或者描述没有提及则进需要关注用户表述内容即可。
- 仓库若为 Monorepo，需确认目标包/应用；默认遵循 `.specify/memory/fe-rule.md` 中记录的应用名称，可通过 `--target <workspace>` 覆盖。

## 目标

在**不偏离既有技术与规范**的前提下，完成用户指定的前端需求实现，包括必要的代码、配置、测试与文档更新；所有改动须严格遵循 `.specify/memory/fe-rule.md` 中的规则与案例。

## 执行步骤

1) 初始化与运行记录定位

   - 从仓库根目录运行 `scripts/powershell/check-prerequisites.ps1 -Json -PathsOnly` 并解析 JSON 输出，获取 `FEATURE_DIR`、`BRANCH`、`PLAN`/`TASKS` 等绝对路径。若参数包含单引号（如 "I'm Groot"），请使用 `'I'\''m Groot'` 或双引号包裹。
   - 设定 `RUNBOOK_DIR = FEATURE_DIR/runbooks`、`RUNBOOK_FILE = RUNBOOK_DIR/fe-rule.md`，确保目录存在；如文件缺失，复制 `.specify/templates/fe-rule-run-template.md` 作为基线文档。
   - 读取现有运行记录内容，准备在后续步骤中增量更新并保留人工补充说明。

2) 解析规则基线

   - 加载 `.specify/memory/fe-rule.md`，提取以下约束并结构化：
     - 技术栈：框架、路由、打包、运行时能力
     - UI/样式：设计体系、原子类/组件库约定、图标集
     - 数据流：HTTP 客户端、全局状态、服务器状态方案
     - 工程规范：目录结构、命名模式、lint/格式化/提交规范
     - 其他约束（可访问性、国际化、性能基线等，如文档中给出）
   - 如发现规则存在空白或冲突，记录为 `未知：[原因]` 并在后续计划中标注需澄清。

3) 获取需求上下文

   - 读取与本次需求关联的说明（优先顺序：用户输入 → `.specify/memory/tasks/*.md` → `tasks.md` → `plan.md` → 其他补充文档），如果相关文件不存在或者描述没有提及则进需要关注用户表述内容即可。
   - 识别本次改动影响的页面/模块/组件，确认依赖关系和潜在风险（如跨包联动、API 协议变更）。
   - 若关键信息缺失（例如接口契约不明确），向用户抛出澄清问题后再继续。

4) 制定实施计划（默认试运行）

   - 列出拟修改/新增的文件、目录与生成脚本，并为每项标注对应的规则章节或原文引用。
   - 说明需新增的依赖或配置调整，并评估对现有规范的影响。
   - 输出详细计划并询问："是否按照该计划继续实施？（是/否）"，在获得确认前可能会有多轮对话进行进一步的方案调整，如果用户给出修改意见你要重新询问是否开始实施？（是/否），在最终确认前不得写入磁盘。

5) 代码实施

   - 按计划逐项执行，保持最小必要改动；对于涉及共享组件或跨包影响的改动，先查阅现有实现以复用模式。
   - 任何新增文件需符合规则中的目录与命名要求（例如 `app/(routes)/`、`components/`、`hooks/`、`lib/` 等）。
   - 引入 UI/状态/HTTP 工具时，使用 `.specify/memory/fe-rule.md` 建议的初始化方式（如 axios 实例、TanStack Query Provider、Zustand store 模板）。
   - 若必须偏离规则（例如第三方依赖已废弃），就差异理由记录在计划或最终报告中，并请求用户确认。

6) 校验与自测

   - 运行规则或仓库推荐的校验脚本（例如 `pnpm lint`、`pnpm test`、`pnpm --filter <app> test`、`pnpm --filter <app> build`）。若环境限制无法执行，给出具体命令供用户手动运行。
   - 对改动涉及的关键路径提供最小测试或故事（React Testing Library、Playwright、Vitest 等），遵循规则中描述的测试策略。
   - 确保所有生成的文件通过格式化与类型检查。

7) 沉淀执行记录

   - 以 `RUNBOOK_FILE` 为目标，按照 `.specify/templates/fe-rule-run-template.md` 的结构更新运行纪要：
     * 在 "基础信息" 中写入当前 `FEATURE_DIR` 的绝对路径与本次运行的更新时间（ISO 8601）。
     * 填充 "## 当前执行摘要"，覆盖需求上下文、依赖范围、遵循的规则章节、核心改动、已执行的验证、遗留风险/待办。
     * 在 "## 变更历史" 顶部新增 `### YYYY-MM-DD HH:mm - <需求概要>` 子节，列出本次涉及的模块、引用的规则章节、关键文件、验证结果与后续动作，保持倒序排列。
   - 保留文件内既有人工注释或过往条目，仅对模板定义的区域执行增量更新。
   - 根据这次运行的对话检查`.specify/memory/fe-figma-gen.md`是否需要修订调整，如果需要请你修订调整（直接修改原来的内容即可）

8) 汇报

   - 汇总改动列表、受影响模块、涉及的规则章节以及已运行/建议运行的校验命令。
   - 提供 `RUNBOOK_FILE` 的绝对路径与本次更新的亮点，提醒团队在后续迭代前阅读该记录。
   - 如果存在未自动完成的步骤（例如需要手动配置 Secrets、更新设计稿），在结尾列出明确的后续动作。

## 注意事项

- 除了最后的修订调整阶段，不得修改或覆盖 `.specify/memory/fe-rule.md`；若规则需更新，单独运行 `/speckit.fe-rule.scan` 或 `/speckit.fe-rule.wizard`。
- 保持幂等性：重复运行同一命令应只更新既定文件，避免重复追加片段。
- 对于无法自动化的高风险改动（数据库迁移、第三方 SDK 升级等），输出建议并请求用户确认后再执行。
- 若仓库存在并行的后端/CLI 改动，请在计划中显式注明前端部分的接口契约假设，防止规则冲突。
