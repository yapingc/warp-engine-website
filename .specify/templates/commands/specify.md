## 用户输入

```text
$ARGUMENTS
```

在继续前，若输入非空，必须先审视用户输入。

## 概览

触发消息中 `/speckit.specify` 之后的文本**即为**功能描述。即便下方出现字面 `{ARGS}`，也假设你在当前对话中始终能获取到它。除非用户输入为空，否则不要要求重复描述。

根据该功能描述执行以下步骤：

1. **生成精简短名**（2–4 个词）供分支使用：
   - 解析功能描述，提取最关键的关键词
   - 创建 2–4 个词的短名，准确概括功能
   - 优先使用动词-名词格式（如 `add-user-auth`、`fix-payment-bug`）
   - 保留技术术语与缩写（OAuth2、API、JWT 等）
   - 保持简洁且便于理解
   - 示例：
     - “我要新增用户认证” → `user-auth`
     - “实现 API 的 OAuth2 集成” → `oauth2-api-integration`
     - “创建分析仪表盘” → `analytics-dashboard`
     - “修复支付超时问题” → `fix-payment-timeout`

2. 从仓库根目录运行 `{SCRIPT}`，并传入上述短名参数，解析 JSON 输出中的 BRANCH_NAME 与 SPEC_FILE。路径必须使用绝对路径。

   **重要提示**：

   - 在 `{SCRIPT}` 命令后追加短名参数（来自步骤 1）
   - Bash：`--short-name "your-generated-short-name"`
   - PowerShell：`-ShortName "your-generated-short-name"`
   - 若参数包含单引号（如 "I'm Groot"），使用 `'I'\''m Groot'` 或 `"I'm Groot"`
   - 此脚本只允许运行一次
   - JSON 输出直接打印在终端，需以此为准获取数据

3. 加载 `templates/spec-template.md`，了解必须填写的章节。

4. 按以下流程执行：

    1. 从输入中解析用户描述  
       若为空：报错 “No feature description provided”
    2. 从描述中提取关键概念  
       识别：参与者、行为、数据、约束
    3. 对不明确的部分：
       - 基于上下文与行业惯例作出合理假设
       - 仅当以下情况才使用 `[NEEDS CLARIFICATION: question]`：
         - 该决策会显著影响范围或用户体验
         - 存在多种合理解释且影响不同
         - 无法给出合理默认值
       - **限制**：最多 3 个 `[NEEDS CLARIFICATION]` 标记
       - 按影响排序：范围 > 安全/隐私 > 用户体验 > 技术细节
    4. 填写 User Scenarios & Testing  
       若无法推导用户流程：报错 “Cannot determine user scenarios”
    5. 生成 Functional Requirements  
       - 每条需求必须可测试
       - 对未指定细节使用合理默认值，并在 Assumptions 中记录
    6. 定义 Success Criteria  
       - 产出可量化、与技术无关的结果
       - 同时包含量化指标（时间、性能、数量）与定性结果（用户满意、任务完成）
       - 所有指标必须在无实现细节的情况下可验证
    7. 识别关键实体（若涉及数据）
    8. 返回：SUCCESS（规格已可用于规划）

5. 使用模板结构将规格写入 SPEC_FILE：保留原有章节顺序与标题，仅用功能描述提炼的具体内容替换占位符。

6. **规格质量验证**：首次写入后，按照以下标准自检：

   a. **创建规格质量检查清单**：在 `FEATURE_DIR/checklists/requirements.md` 中生成清单（遵循模板结构），包含以下内容：
   
      ```markdown
      # 规格质量检查清单： [FEATURE NAME]
      
      **Purpose（用途）**: 在进入规划前验证规格的完整性与质量
      **Created（创建时间）**: [DATE]
      **Feature（规格路径）**: [Link to spec.md]
      
      ## 内容质量
      
      - [ ] 未包含实现细节（语言、框架、API 等）
      - [ ] 聚焦用户价值与业务诉求
      - [ ] 适合非技术干系人阅读
      - [ ] 所有必填章节均已完成
      
      ## 需求完备性
      
      - [ ] 不存在未解决的 [NEEDS CLARIFICATION] 标记
      - [ ] 需求可测试且无歧义
      - [ ] 成功标准可量化
      - [ ] 成功标准与技术无关（未出现实现细节）
      - [ ] 验收场景齐备
      - [ ] 边界情况已识别
      - [ ] 范围界定清晰
      - [ ] 依赖与假设已列出
      
      ## 功能准备度
      
      - [ ] 所有功能需求都具备明确的验收标准
      - [ ] 用户场景覆盖主要流程
      - [ ] 满足成功标准中设定的可量化结果
      - [ ] 规格中未渗透实现层细节
      
      ## Notes（备注）
      
      - 未完成的条目需在运行 `/speckit.clarify` 或 `/speckit.plan` 前更新规格
      ```
   
   b. **执行校验**：逐项检查规格：
      - 判断每项是否通过
      - 如发现问题，引用规格相关段落记录
   
   c. **处理校验结果**：
      
      - **全部通过**：标记清单完成，进入步骤 6d
      
      - **存在失败项（不含 `[NEEDS CLARIFICATION]`）**：
        1. 列出失败项及具体问题
        2. 更新规格以解决问题
        3. 重新校验，最多 3 轮
        4. 若 3 轮后仍未通过，在清单备注中记录剩余问题并警示用户
      
      - **仍存在 `[NEEDS CLARIFICATION]`**：
        1. 提取规格里的所有 `[NEEDS CLARIFICATION: ...]`
        2. **数量检查**：如超过 3 个，仅保留最关键的 3 个（优先范围/安全/UX），其余使用合理假设补齐
        3. 对仍需澄清的每个问题（最多 3 个），按以下格式向用户呈现选项：
        
           ```markdown
           ## Question [N]： [Topic]
           
           **Context（上下文）**: [引用规格中的相关段落]
           
           **What we need to know（需要确认）**: [来自 NEEDS CLARIFICATION 标记的具体问题]
           
           **Suggested Answers（建议答案）**:
           
           | Option | Answer       | Implications         |
           | ------ | ------------ | -------------------- |
           | A      | [建议答案 1] | [对特性的影响]       |
           | B      | [建议答案 2] | [对特性的影响]       |
           | C      | [建议答案 3] | [对特性的影响]       |
           | Custom | 自定义答案   | [如何提供自定义输入] |
           
           **Your choice（选择）**: _[等待用户回答]_
           ```
        
        4. **关键 - 表格格式**：确保 Markdown 表格渲染正确：
           - 竖线与内容之间保留空格，例如 `| Content |`
           - 表头分隔线至少 3 个连字符：`|--------|`
           - 在 Markdown 预览中确认表格正常显示
        5. 问题顺序编号（Q1、Q2、Q3，最多 3 个）
        6. 所有问题一次性展示后再等待回复
        7. 等待用户按格式答复（如 “Q1: A, Q2: Custom - ... , Q3: B”）
        8. 将用户选择或自定义答案写入规格，替换对应的 `[NEEDS CLARIFICATION]`
        9. 所有澄清解决后重新执行校验
   
   d. **更新清单**：每轮校验后同步更新清单的通过/失败状态。

7. 汇报完成情况：包括分支名、规格文件路径、检查清单结果，以及是否已准备好进入下一阶段（`/speckit.clarify` 或 `/speckit.plan`）。

**注意**：脚本会创建并检出新分支，并在写入前初始化规格文件。

## 一般准则

## 快速指引

- 聚焦用户需要的 **是什么（WHAT）** 与 **为什么（WHY）**
- 避免涉及实现方式（不写技术栈、API、代码结构）
- 面向业务干系人而非开发者
- **不要**在规格内嵌任何检查清单，相关工作交由其他命令完成

### 章节要求

- **必填章节**：每个功能都必须完成
- **可选章节**：仅在功能相关时保留
- 若章节不适用，直接移除（不要保留 “N/A”）

### 面向 AI 生成的建议

当依据用户提示生成规格时：

1. **作出合理假设**：结合上下文、行业标准、常见模式填补空缺
2. **记录假设**：在 Assumptions 中记下已采用的默认值
3. **限制澄清**：最多 3 个 `[NEEDS CLARIFICATION]`，仅用于关键决策：
   - 对功能范围或用户体验影响重大
   - 多种合理解释导致后果不同
   - 无任何合理默认可选
4. **澄清优先级**：范围 > 安全/隐私 > 用户体验 > 技术细节
5. **站在测试者视角**：任何含糊需求都应被视为未通过 “可测试且明确” 条目
6. **常见需要澄清的方面**（仅当缺乏合理默认）：
   - 功能范围与边界（哪些使用场景纳入/排除）
   - 用户类型与权限（存在多种可能解释时）
   - 安全/合规要求（涉及法律或重要财务影响）

**可接受的合理默认值**（无需询问）：
- 数据保留：遵循该领域行业惯例
- 性能指标：采用标准 Web/移动应用期望
- 错误处理：提供友好的提示与适当回退
- 认证方式：Web 应用默认基于 Session 或 OAuth2
- 集成模式：默认 RESTful API，除非另有说明

### 成功标准指引

成功标准必须满足：

1. **可量化**：包含具体指标（时间、比例、次数、速率）
2. **与技术无关**：不提及框架、语言、数据库或工具
3. **面向用户**：从用户/业务角度描述结果，而非系统内部
4. **可验证**：无需了解实现细节即可测试/验证

**优秀示例**：
- “用户在 3 分钟内完成结账”
- “系统可同时支持 10,000 名在线用户”
- “95% 的搜索在 1 秒内返回结果”
- “任务完成率提升 40%”

**不佳示例**（过于聚焦实现细节）：
- “API 响应时间低于 200ms”（过于技术化，可改写为 “用户几乎即时看到结果”）
- “数据库可处理 1000 TPS”（实现细节，可转换为用户体验指标）
- “React 组件渲染效率高”（限定具体框架）
- “Redis 缓存命中率高于 80%”（特定技术指标）
