## 用户输入

```text
$ARGUMENTS
```

请提供接口 ID、接口用途说明、期望的调用场景/页面，以及需要特别关注的数据字段或交互约束。

## 前置条件

- `.specify/memory/fe-rule.md` 为最新版本，包含项目的目录、命名、网络层与类型定义约束。
- 已安装可通过接口 ID 获取完整接口定义的工具（默认命令中调用，无需重复配置）。
- 目标工作区具备构建与测试脚本，遵循 fe-rule 中的约束。

## 目标

从指定的接口 ID 获取权威的请求/响应定义，在遵循 fe-rule 规范的前提下生成或更新：接口声明、类型定义、网络层封装以及相关页面/组件逻辑，并沉淀运行记录以便后续增量更新。

## 执行步骤

1) **初始化与运行记录定位**
   - 从仓库根目录运行 `scripts/powershell/check-prerequisites.ps1 -Json -PathsOnly` 并解析 JSON，获取 `FEATURE_DIR`、`BRANCH`、`IMPL_PLAN`、`TASKS` 的绝对路径。若参数包含单引号（如 "I'm Groot"），请使用 `'I'\''m Groot'` 或双引号包裹。
   - 定义 `RUNBOOK_DIR = FEATURE_DIR/runbooks` 与 `RUNBOOK_FILE = RUNBOOK_DIR/fe-definition-gen.md`（绝对路径）。如文件不存在，先创建目录并复制 `.specify/templates/fe-definition-gen-run-template.md` 作为初始内容。
   - 读取既有运行记录，保留历史条目以便后续增量更新。

2) **加载工程约束**
   - 解析 `.specify/memory/fe-rule.md`，提取：网络层封装模式（fetch/axios/自定义 client）、目录约定、类型文件存放位置、组件命名规则、测试策略等。
   - 若 fe-rule 中定义了默认的接口声明文件、类型聚合文件或请求工厂，记下具体路径以便后续写入。

3) **解析输入并确认信息缺口**
   - 从 `$ARGUMENTS` 中提取接口 ID、调用场景、目标页面/组件、预期的交互行为与数据展示重点。
   - 若缺失关键信息（例如接口 ID 或页面入口），应先向用户确认后再继续。

4) **获取接口定义**
   - 调用项目提供的接口定义获取工具（例如 `fuxi-definition` tool），传入接口 ID 并请求完整的请求/响应 schema、错误码、示例 payload 与鉴权要求。
   - 若工具返回告警或缺失字段，记录在临时列表中，必要时提示用户补充后再继续。

5) **对齐接口与类型约束**
   - 先确认当前请求声明是否已经存在该接口，存在的话在原来已经有的基础上比对最新的接口声明做修改。
   - 在 fe-rule 指定的接口声明位置追加或更新该接口的请求定义，确保方法、路径、参数说明、错误处理策略与返回值均符合工具输出。
   - 在 fe-rule 指定的类型/`types` 目录中生成或更新对应的 TypeScript 定义（请求参数、响应数据、错误结构），命名遵循项目约定。
   - 若需要新建共享类型或枚举，先检查是否已有同名导出，避免重复定义。

6) **生成调用封装与集成计划**
   - 根据接口定义与 fe-rule 指南，规划网络层封装（service/hooks）和缓存/状态策略，标注需修改或新增的文件。
   - 结合用户描述的页面/组件上下文，列出需要更新的 UI 区域、状态流转、边界处理与跟进的测试项。
   - 输出详细计划并询问：“是否根据以上计划开始实施？（是/否）”，在获得确认前可能会有多轮对话进行进一步的方案调整，如果用户给出修改意见你要重新询问是否开始实施？（是/否），在最终确认前不得写入磁盘。

7) **实施接口封装与类型定义**
   - 按计划先落地接口封装（service、hook、client），保证请求参数校验、错误处理、重试/缓存策略符合 fe-rule。
   - 将工具返回的示例或约束写入 JSDoc/注释，便于后续维护。
   - 项目存在测试代码逻辑则更新或新增相关测试（单元或集成），覆盖成功、失败与边界分支。

8) **更新页面与组件逻辑**
   - 依据用户场景在指定页面/组件中注入新的数据请求、状态处理与 UI 呈现，遵循 fe-rule 的目录与命名规范。
   - 若需要组件或样式调整，优先复用项目共享资源；避免引入未授权依赖。
   - 项目存在测试代码逻辑则对新增交互提供最小的可验证测试（React Testing Library / Playwright 等），并在必要时记录人工验证点。

9) **校验与自测**
   - 运行推荐的校验命令（如 `pnpm lint`、`pnpm test --filter <workspace>`、`pnpm --filter <workspace> build`），若受限则给出需人工执行的命令及预期结果。
   - 确保类型检查通过，网络层及 UI 组件引用的共享模块存在且版本兼容。

10) **沉淀执行记录**
   - 使用 `RUNBOOK_FILE` 更新 `.specify/templates/fe-definition-gen-run-template.md` 定义的结构：
     * 同步“基础信息”中的功能目录（`FEATURE_DIR` 绝对路径）与最新更新时间（ISO 8601）。
     * 在“## 当前执行摘要”中记录接口 ID、关键接口字段、生成/更新的类型与封装、影响的页面/组件、验证状态、遗留问题。
     * 在“## 历史记录”顶部追加 `### YYYY-MM-DD HH:mm - <接口 ID>` 的条目，包含本次输入摘要、改动文件、验证命令、后续动作，保持最新记录置顶。
   - 保留 runbook 中既有手动备注，仅在模板区域增量更新。

11) **汇报**
   - 汇总新增/修改的文件（类型、服务、组件、测试）、已执行的校验命令以及 `RUNBOOK_FILE` 路径。
   - 强调后续需要关注的风险或待办事项，并建议下一步操作（如提交前需执行的命令、向后端同步的契约变更等）。

## 注意事项

- 严格遵循 `.specify/memory/fe-rule.md` 的目录、命名与依赖约束；如需偏离，必须在计划与汇报中说明原因。
- 所有新增类型与接口必须具备明确来源；若工具返回的数据与 fe-rule 冲突，优先寻求用户澄清。
- 维护幂等性：重复运行本命令应复用既有封装与 runbook 记录，避免重复追加。
- 若接口定义工具返回错误或缺失字段，请先解决问题或记录阻塞后再继续实施。
