## 用户输入

```text
$ARGUMENTS
```

在继续前，若输入非空，必须先审视用户输入。

## 概览

目标：发现并消除当前功能规格里的歧义或缺失决策节点，并将澄清结果直接记录到规格文件中。

注意：澄清流程应当在调用 `/speckit.plan` 之前完成。若用户明确表示跳过澄清（例如探索性 Spike），可以继续执行，但必须提醒下游返工风险会提高。

执行步骤：

1. 从仓库根目录运行 `{SCRIPT}` **一次**（合并使用 `--json --paths-only` / `-Json -PathsOnly`）。解析 JSON 最小字段：
   - `FEATURE_DIR`
   - `FEATURE_SPEC`
   - （可选）`IMPL_PLAN`、`TASKS` 供后续链式流程使用
   - 若解析失败，立即终止并提示用户重新运行 `/speckit.specify` 或检查特性分支环境
   - 当参数中包含单引号（如 "I'm Groot"）时，需使用 `'I'\''m Groot'` 或双引号 `"I'm Groot"`

2. 加载当前规格文件。依据下列分类执行结构化歧义/覆盖扫描，并为每个类别标记状态：Clear / Partial / Missing。生成用于优先级排序的内部覆盖地图（除非无需提问，否则不要直接输出）。

   Functional Scope & Behavior（功能范围与行为）：
   - 核心用户目标与成功标准
   - 明确的 out-of-scope 声明
   - 用户角色 / 画像区分

   Domain & Data Model（领域与数据模型）：
   - 实体、属性、关系
   - 标识与唯一性规则
   - 生命周期 / 状态变迁
  - 数据量级 / 规模假设

   Interaction & UX Flow（交互与体验流程）：
   - 关键用户旅程 / 序列
   - 错误、空态、加载状态
   - 无障碍或本地化说明

   Non-Functional Quality Attributes（非功能质量属性）：
   - 性能（延迟、吞吐目标）
   - 可扩展性（横向/纵向、容量上限）
   - 可靠性与可用性（可用率、恢复预期）
   - 可观测性（日志、指标、链路追踪信号）
   - 安全与隐私（认证/授权、数据保护、威胁假设）
   - 合规 / 法规约束（如适用）

   Integration & External Dependencies（集成与外部依赖）：
   - 外部服务 / API 及其失败模式
   - 数据导入 / 导出格式
   - 协议 / 版本假设

   Edge Cases & Failure Handling（边界场景与失败处理）：
   - 负面场景
   - 限流 / 节流
   - 冲突解决（如并发编辑）

   Constraints & Tradeoffs（约束与取舍）：
   - 技术约束（语言、存储、托管）
   - 明确的取舍或被拒绝方案

   Terminology & Consistency（术语与一致性）：
   - 术语表中的标准用语
   - 需避免的同义词 / 弃用术语

   Completion Signals（完成信号）：
   - 验收标准的可测试性
   - 可量化的 “Definition of Done” 指标

   Misc / Placeholders（杂项 / 占位符）：
   - TODO 标记 / 未决事项
   - 缺乏量化的模糊形容词（如 “robust”“intuitive”）

   若某分类状态为 Partial 或 Missing，需生成候选问题，除非：
   - 澄清不会对实施或验证策略产生实质影响
   - 信息更适合在规划阶段解决（内部记录即可）

3. 在内部生成优先级队列（最多 5 个候选澄清问题），勿一次性全部输出。遵循：
    - 全程累计提问不超过 10 个。
    - 每个问题必须能通过以下任一方式回答：
       * 多选题（2–5 个互斥选项）
       * 限制长度的短答（显式要求 “Answer in <=5 words”）
    - 仅保留那些回答会显著影响架构、数据建模、任务拆解、测试设计、交互行为、运维准备或合规验证的问题。
    - 保持分类均衡：优先覆盖影响最大的未解决类别，避免在存在高影响缺口时去问两个低价值问题。
    - 排除已回答、纯风格偏好或纯规划执行层面的题目（除非阻碍正确性）。
    - 优先选择能降低下游返工或避免验收标准错位的澄清。
    - 若未解决类别超过 5 个，按（Impact * Uncertainty）评分取前 5 个。

4. 顺序提问循环（交互式）：
    - 每次仅呈现 **一个** 问题。
    - 若为多选题：
       * **评估所有选项**，依据以下因素挑出**最优选项**：
          - 针对项目类型的最佳实践
          - 相似实现中的常见模式
          - 风险降低（安全、性能、可维护性）
          - 与规格中明确目标或约束的一致性
       * 将推荐项置于开头突出显示：`**Recommended:** Option [X] - <理由>`（1–2 句中文说明）
       * 将全部选项以 Markdown 表格列出：

       | Option | Description           |
       | ------ | --------------------- |
       | A      | <选项 A 描述>         |
       | B      | <选项 B 描述>         |
       | C      | <选项 C 描述>         | （可按需扩展至 D/E） |
       | Short  | 自定义短答（≤5 个词） | （仅在适用时出现）   |

       * 表格后补充提示：`可回复选项字母（如 "A"），若接受推荐项可回复 "yes" 或 "recommended"，也可提供自定义短答。`
    - 若为短答题：
       * 先给出基于上下文的**建议答案**
       * 格式：`**Suggested:** <建议答案> - <简短理由>`
       * 随后说明：`格式：不超过 5 个词的短答。若接受建议，可回复 "yes" 或 "suggested"，也可输入自定义答案。`
    - 用户回答后：
       * 若回复 “yes”“recommended” 或 “suggested”，沿用先前给出的推荐/建议
       * 否则验证答案是否对应某选项或满足 ≤5 个词的限制
       * 若表述含糊，需再行澄清（仍计为同一道问题，不前进）
       * 答复确定后记录在工作记忆（暂不写盘），继续下一个问题
    - 满足以下任一条件即停止提问：
       * 关键歧义已提前解决，剩余问题无必要
       * 用户表示结束（“done”“good”“no more” 等）
       * 已提问 5 个
    - 禁止提前泄露队列中尚未询问的问题。
    - 若一开始就没有有效问题，直接报告不存在需要正式澄清的关键歧义。

5. 每接受一条答案后立即整合（渐进式写入）：
    - 保持规格的内存表示与原始内容
    - 首次整合本轮答案时：
       * 确保存在 `## Clarifications` 章节（如缺失，按模板在最高级上下文/概述章节之后创建）
       * 在其下创建（若不存在）`### Session YYYY-MM-DD` 子标题（日期为当天）
    - 每次确认答案后追加条目：`- Q: <question> → A: <final answer>`
    - 随即将澄清内容更新到最合适的章节：
       * 功能歧义 → 更新或新增 Functional Requirements 中的条目
       * 交互 / 角色区分 → 更新 User Stories 或 Actors 子章节（若存在）
       * 数据形态 / 实体 → 在 Data Model 中新增字段、类型、关系并保持顺序，简洁记录约束
       * 非功能约束 → 在 Non-Functional / Quality Attributes 中新增或修改可量化指标，将模糊形容词转化为具体目标
       * 边界场景 → 在 Edge Cases / Error Handling 下新增条目（或按模板创建子章节）
       * 术语冲突 → 在规格中统一术语，如需保留旧称，仅在首次出现时标注 `(formerly referred to as "X")`
    - 若澄清结果使既有含糊陈述失效，应直接替换而非重复；务必清除互相矛盾的旧描述
    - 每次写入后立即保存规格文件（原子覆盖），降低上下文丢失风险
    - 保持排版：勿随意重排无关章节；维持标题层级
    - 每次新增内容应简洁且可验证，避免散文化

6. 校验（每次写入后及最终）：
   - Clarifications 章节中每条答案恰好对应一行（不得重复）
   - 累计提问并被接受的数量 ≤ 5
   - 更新的章节中不存在澄清本应解决却残留的模糊占位
   - 不得保留相互矛盾的旧表述
   - Markdown 结构有效；仅允许新增 `## Clarifications` 与 `### Session YYYY-MM-DD`
   - 术语一致：整份规格中使用统一的标准术语

7. 将更新后的规格写回 `FEATURE_SPEC`。

8. 汇报完成（提问循环结束或提前终止后）：
   - 已提问并回答的问题数量
   - 更新后的规格路径
   - 被修改的章节名称
   - 覆盖率汇总表：列出每个分类的状态——Resolved（原为 Partial/Missing 且已解决）、Deferred（超出问题配额或更适合在规划阶段处理）、Clear（原本就充分）、Outstanding（仍为 Partial/Missing 但影响较低）
   - 若存在 Outstanding 或 Deferred，建议是继续 `/speckit.plan` 还是待规划后再次运行 `/speckit.clarify`
   - 推荐下一步命令

行为准则：
- 若未发现具备实质影响的歧义（或剩余问题影响甚微），可答复 “No critical ambiguities detected worth formal clarification.” 并建议继续后续流程
- 若缺少规格文件，提示先运行 `/speckit.specify`（此处不要创建新规格）
- 总提问数不得超过 5（同一题的澄清不另计）
- 除非影响功能清晰度，否则避免猜测技术栈
- 尊重用户的提前结束信号（“stop”“done”“proceed” 等）
 - 若因覆盖面充足而无需提问，应输出精简覆盖摘要（全部 Clear），随后建议继续
 - 若达到提问配额但仍有高影响类别未解，需在 Deferred 中明确标注原因

优先级上下文：{ARGS}
