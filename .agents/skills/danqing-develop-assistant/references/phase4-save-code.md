# Phase 4: 保存代码

当用户说"保存代码"、"备份一下"、"同步到云端"等类似表达时：

1. 确认远程仓库已创建（若未创建，先调用 `gitlab_repo(action="create", app_id="xxx", name="xxx")` 创建）
2. 系统性收集项目文件并推送到远程仓库
3. 完成后告诉用户：
   > 代码已保存到云端，不用担心丢失了。

---

## 推送流程（NON-NEGOTIABLE）

使用 `gitlab_repo(action="push")` 通过 Commits API 推送，**必须**按以下步骤完整执行：

### 步骤 1：获取远程仓库 project_id

如果是首次推送，从 `gitlab_repo(action="create")` 的返回结果中获取 `project_id`。
如果仓库已存在，从 `gitlab_repo(action="list", search="xxx")` 中获取。

### 步骤 2：系统性收集全部项目文件（最关键的一步）

**禁止只传"本次会话中读过的文件"。必须完整遍历项目目录，收集所有源代码文件。**

具体做法：

1. 用 Glob 工具列出项目内所有文件：`**/*`
2. **排除以下目录/文件**（不推送）：
   - `node_modules/`
   - `.git/`
   - `dist/`、`build/`
   - `.env`、`.env.local`（含密钥，禁止推送）
   - `old/`（备份目录）
   - 其他构建产物或临时文件
3. 对列出的每个文件，用 Read 工具读取其完整内容
4. 组装为 `{"文件相对路径": "文件内容"}` 格式的对象

**自检**：组装完成后，核对文件数量是否与 Glob 列出的（排除后的）数量一致。如果明显少于预期，说明遗漏了文件，必须补全。

### 步骤 3：执行推送

```
gitlab_repo(action="push", project_id=项目ID, branch="master", files=收集到的全部文件, commit_message="保存代码")
```

### 步骤 4：验证推送结果

推送完成后，检查返回结果：
- 如果返回成功，确认提交的文件数量与预期一致
- 如果返回错误，排查原因并重试
- **禁止在未确认推送成功的情况下告诉用户"代码已保存"**

---

## 增量推送（非首次保存）

如果远程仓库已有代码（非首次推送），只需推送变更文件即可：

1. 确认哪些文件是新增或修改过的（根据本次会话中的操作记录判断）
2. 用 Read 读取这些文件的最新内容
3. 调用 `gitlab_repo(action="push", project_id=项目ID, files=变更文件)` 推送
4. 如果不确定哪些文件变更了，按首次推送的完整收集流程执行（宁可多推不可漏推）

---

## 远程仓库配置

如果远程仓库未创建：
1. 调用 `gitlab_repo(action="create", app_id="xxx", name="xxx")` 创建
2. 记录返回的 `project_id`，后续推送使用

如果需要获取已有仓库信息：
- 用 `gitlab_repo(action="clone_url", app_id="xxx")` 获取仓库地址（用于展示给用户或配置本地 remote）
- 用 `gitlab_repo(action="list", search="xxx")` 查找仓库 ID

| 工具 | 用途 | 向用户的说法 |
|------|------|------------|

| `join_app(invite_code)` | 使用邀请码加入 | "请把收到的邀请码发给我" |
| `clone_project(app_id)` | 克隆已有项目 | "我来帮您获取这个项目的代码" |
